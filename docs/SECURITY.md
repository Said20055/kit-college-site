# Безопасность и соответствие 152-ФЗ

Документ описывает реализованные меры защиты сайта ГБПОУ РД «КИТ» и чек-лист
перед публичным запуском.

## Аутентификация и доступ

- **Пароли** — Argon2id (`hash-wasm`, m=19456, t=2, p=1), без хранения в открытом виде.
- **Сессии** — серверные: в cookie только случайный токен (`httpOnly`, `secure`
  в проде, `SameSite=Lax`), в БД хранится его SHA-256; TTL 8 часов.
- **Защита от подбора** — блокировка учётной записи после 5 неудачных попыток на
  15 минут; для входа применяется «dummy»-хеш для выравнивания времени ответа.
- **RBAC** — роли `ADMIN`/`EDITOR`. Проверка в DAL (`lib/auth/dal.ts`) и повторно
  в каждом Server Action; `proxy.ts` — лишь оптимистичный предварительный фильтр.
- **Аудит** — журнал входов и изменений контента/пользователей (`/admin/audit`),
  с IP и User-Agent (из `X-Forwarded-For`, который проставляет nginx).
- **2FA** — зарезервировано в схеме (`totpSecret`/`totpEnabled`), включается позже.

## Заголовки безопасности (`next.config.ts`)

`Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`,
`Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy`,
`Cross-Origin-Opener-Policy: same-origin`, `X-DNS-Prefetch-Control: off`.
`Strict-Transport-Security` добавляется nginx (TLS-терминация). Заголовок
`X-Powered-By` отключён (`poweredByHeader: false`).

**CSP — обоснование выбора.** Сознательно используется статически совместимый
вариант `script-src 'self' 'unsafe-inline'` вместо nonce/`strict-dynamic`:
по документации Next nonce принудительно переводит все страницы в динамический
рендер и отключает статическую генерацию и ISR, на которых держится требование
«правки видны без передеплоя». Сторонние скрипты и домены не подключаются, поэтому
поверхность XSS минимальна. Если в будущем потребуется строгая CSP без
`unsafe-inline` при сохранении кэширования — рассмотреть экспериментальный
SRI (hash-based) из документации Next.

## Загрузка и отдача файлов

- Файлы хранятся **вне webroot** (`MEDIA_DIR`, том `kit-uploads`) и отдаются
  только через маршрут `/media/[id]` с корректными заголовками.
- Белый список MIME (изображения: jpeg/png/webp; документы: pdf/doc/docx/xls/xlsx),
  лимит размера 25 МБ, защита от path traversal (`path.basename`), имя файла на
  диске генерируется из контрольной суммы.
- Тело новости (единственный пользовательский HTML) проходит `sanitize-html` по
  строгому allowlist тегов/атрибутов перед сохранением.

## Сеть и инфраструктура

- TLS 1.2/1.3 на nginx, редирект HTTP→HTTPS, HSTS.
- PostgreSQL опубликован только на `127.0.0.1` (для сборки и бэкапов), извне закрыт.
- Веб-приложение работает под непривилегированным пользователем в контейнере.
- Регулярные бэкапы БД и загрузок с ротацией (`scripts/backup.sh`).

- [x] Первичная запись/хранение/уточнение ПДн в БД на территории РФ (ст. 18 ч. 5).
- [x] Минимизация ПДн: собираются только данные сотрудников для раздела «Сведения»

- [x] Защищённая передача (TLS) и хранение паролей (Argon2id).
- [x] Резервное копирование и процедура восстановления.

      подать уведомление в Роскомнадзор.

## Перед публичным запуском

1. Сменить пароль администратора и все пароли из `.env` на стойкие; не хранить
   `.env` в репозитории.
2. Загрузить реальные документы в медиатеку и привязать к разделу «Документы».
3. Проверить микроразметку раздела «Сведения» **официальным парсером
   Рособрнадзора** (Приказ № 1493) на боевом домене.
4. Установить реальные TLS-сертификаты, настроить автопродление.
5. (Опционально) Ограничить доступ к `/admin` по IP на уровне nginx и/или
   включить 2FA, когда будет реализована.
6. Проверить доступность (ГОСТ Р 52872-2019, WCAG AA) и корректность `sitemap.xml`,
   `robots.txt`, JSON-LD после наполнения контентом.
